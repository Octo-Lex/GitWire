# Sidecar Container-Runtime Executor (v0.23.0)

## Status

Design pass. Task 0 (CT 115 feasibility) is GREEN — proceeding with the
sidecar/container-runtime direction. No implementation code yet; this document
defines the contract for review before Task 2+.

Companion to: `docs/architecture/validator-execution-model.md` (Gap 1, v0.22.0).

## Context

v0.22.0 made the validator execution topology honest. On CT 115 it now reports:

```text
executor.selected_kind          = local-process
executor.selected_pass_capable  = false
validator.pass_capable          = false
validator.reason                = selected_backend_not_pass_capable
```

This is correct but not the end goal: CT 115 cannot *produce* pass-capable
validator evidence. `local-process` is structurally non-pass-capable, and the
existing `docker-executor` backend — while `supports_pass: true` — is not
safely reachable from the app container on CT 115 (the app process should not
own Docker authority; binding the host socket into the app container is
explicitly disallowed by the Gap 1 decision).

v0.23.0 introduces a **sidecar executor**: a separate container in the compose
network that owns container-runtime access and runs the pinned validator image
on behalf of the app. The app calls a narrow private API; it never touches
Docker directly.

## Task 0 feasibility result (CT 115, 2026-06-22)

| Check | Result |
|---|---|
| Docker available | ✓ Docker 29.5.0 (inside CT 115, not just on the Proxmox host) |
| Kernel / cgroup | ✓ Proxmox 6.8.12-13-pve kernel, cgroup v2, systemd |
| Storage driver | ✓ overlayfs |
| Resource headroom | ✓ 16GB total / ~15GB available; 9 current containers use ~280MB |
| Isolated container runs | ✓ alpine:3.20 with full validator flags exited 0 |
| `--network=none` enforced | ✓ outbound DNS blocked (not just empty network) |
| `--read-only` enforced | ✓ rootfs write rejected |
| non-root enforced | ✓ uid=1000 |
| pids/memory/tmpfs limits | ✓ accepted + tmpfs writable |
| Dangerous privileges required | **none** — standard Docker flags only |

**Decision: PROCEED with sidecar/container-runtime.** No pivot to delegated-run.
No privilege narrowing needed.

### Key finding: Docker-in-LXC already works

Docker runs *inside* CT 115 (the LXC container), using the Proxmox host's
kernel via cgroup v2 delegation. This means the sidecar does **not** need a
host Docker socket mount (`/var/run/docker.sock`) — it can either run its own
`dockerd` or reach the existing CT-115 Docker daemon over the compose network.
This is a stronger isolation boundary than the originally-proposed socket
mount and is the preferred shape.

## Decision

```text
gitwire-app
  └── calls sidecar's private HTTP API over the docker-compose network
  └── constructs the final GitWire execution receipt from sidecar evidence
  └── holds NO Docker authority

gitwire-executor-sidecar
  └── owns Docker/Podman access (its own dockerd, or the CT-115 daemon)
  └── runs ONLY the pinned validator image
  └── accepts ONLY allowlisted command IDs (no shell, no arbitrary argv)
  └── enforces network=none, read-only rootfs, non-root, resource limits
  └── returns bounded execution evidence
  └── holds NO DB access, NO GitHub credentials, NO repo write access
```

### Why sidecar over delegated-run (for v0.23.0)

- **Smallest step from the existing model.** `dockerExecutorBackend` already
  implements the isolation contract (`--network=none`, `--read-only`,
  `--user=1000:1000`, pids/memory/tmpfs limits). The sidecar wraps that
  execution model behind a private API rather than inventing a new one.
- **No vendor lock-in.** Delegated-run (option B) commits to a provider
  contract (E2B/Modal/etc.) that's harder to walk back. Sidecar keeps
  execution in-house and defers the multi-tenant/cloud decision.
- **Unblocks the Gap 1 promise.** v0.22.0 made topology honest; the sidecar
  is what makes `validator.pass_capable = true` achievable on CT 115.

Delegated-run remains the fallback if a future environment cannot host a
sidecar. The `delegatedRunProvider.js` contract from Gap 1 Phase 4 stays in
place untouched.

## Sidecar API contract

The sidecar exposes exactly two endpoints. It is deliberately **not** a
general Docker API — callers cannot name arbitrary images or run arbitrary
commands.

### `GET /health`

```json
{
  "status": "ok",
  "sidecar_executor_id": "sidecar-executor",
  "sidecar_version": "1.0.0",
  "sidecar_instance_id": "<random per-instance uuid>",
  "container_runtime": "docker",
  "runtime_version": "29.5.0",
  "validator_image_ref": "<configured>",
  "validator_image_digest": "<configured>",
  "ready": true
}
```

`ready` is true only when: runtime reachable, validator image identity
configured, and image inspection has succeeded at least once. Used by the
app's reachability probe (Task 4).

### `POST /v1/validate`

**Authenticates** via a shared bearer token (`GITWIRE_EXECUTOR_SIDECAR_TOKEN`)
set identically in both containers' environments. Not a substitute for the
network boundary — the sidecar must only be reachable on the private compose
network, never exposed externally.

**Request:**

```json
{
  "request_id": "<uuid, for tracing>",
  "files": [
    { "path": "src/foo.js", "content": "..." }
  ],
  "commands": ["lint", "test", "build", "typecheck"],
  "limits": {
    "wall_clock_ms": 30000,
    "memory_mb": 512,
    "pids_limit": 64,
    "output_bytes": 1048576
  },
  "validator_image_ref": "<registry/repo/image@sha256:...>",
  "validator_image_digest": "sha256:...",
  "expected_executor_policy": {
    "network_disabled": true,
    "non_root": true,
    "read_only_rootfs": true,
    "resource_limits": true
  }
}
```

**Response (success path):**

```json
{
  "sidecar_executor_id": "sidecar-executor",
  "sidecar_version": "1.0.0",
  "sidecar_instance_id": "<uuid>",
  "container_runtime": "docker",
  "runtime_version": "29.5.0",
  "validator_image_ref": "<full ref>",
  "validator_image_digest": "sha256:...",
  "inspected_image_digest": "sha256:...",
  "inspection_hash": "sha256:...",
  "network_disabled": true,
  "non_root": true,
  "read_only_rootfs": true,
  "resource_limits": { ... },
  "command_results": [
    {
      "command": "lint",
      "exit_status": 0,
      "output_ref": "output:sha256:...",
      "output_hash": "sha256:...",
      "duration_ms": 1234
    }
  ],
  "aggregate_exit_status": 0,
  "overall": "pass",
  "sidecar_report_hash": "sha256:<over the full response object minus this field>"
}
```

**Response (failure / inconclusive paths):**

```json
{
  "overall": "inconclusive",
  "inconclusive_reason": "validator_image_ref_mismatch | image_inspection_failed | runtime_unreachable | execution_incomplete | executor_error",
  "inconclusive_detail": "...",
  "sidecar_executor_id": "sidecar-executor",
  "sidecar_version": "1.0.0"
}
```

The sidecar never returns `overall: "pass"` unless it has actually run all
commands against the pinned image AND every isolation flag was enforced AND
the inspected image digest matches the configured digest. The app re-verifies
this (defense in depth — see Receipt Binding below).

### `sidecar_report_hash` semantics

The sidecar computes a content-addressed hash over its full response object
(minus the hash field itself), exactly mirroring GitWire's execution-receipt
pattern. The app binds this hash into the GitWire execution receipt so the
verifier gate can confirm the receipt was built from a specific, unmodified
sidecar report — not an arbitrary response the app constructed on its own.

## Security boundary

The sidecar is privileged *relative to the app* (it can create containers),
so its API must be narrower than Docker. Enforced invariants:

```text
Allowed:
  - run ONLY the configured validator image (request ref must match config)
  - mount ONLY the disposable workspace provided in the request
  - execute ONLY allowlisted command IDs (resolved via argv templates)
  - enforce network=none, read-only rootfs, non-root, resource limits
  - return evidence

Forbidden (by design, enforced in sidecar code):
  - arbitrary image names (validator_image_ref must equal configured)
  - arbitrary shell commands (commands[] must be in COMMAND_TEMPLATES)
  - DB access (sidecar has no DATABASE_URL)
  - GitHub credentials (sidecar has no GITHUB_* env)
  - repo write access (sidecar has no repo mounts)
  - outbound network inside the validator container (--network=none)
```

This mirrors the existing `dockerExecutorBackend` isolation contract; the
sidecar is essentially `dockerExecutorBackend` re-homed behind a private API.

### Network exposure

The sidecar MUST NOT be exposed outside the compose network. Concretely:
- No port published to the host (no `ports:` mapping, only `expose:`).
- The Cloudflare tunnel (`gitwire-tunnel-1`) routes only to `gitwire-app`,
  never to the sidecar.
- The shared bearer token is a second layer of defense, not the primary one.

### Receipt binding (the load-bearing invariant)

The app constructs the final GitWire execution receipt from the sidecar's
evidence — the sidecar does not write receipts. The receipt must bind:

```text
execution_backend_id     = "sidecar-executor"
executor_kind            = "container-runtime"
executor_pass_capable    = true   (only when all four conditions hold)
validator_image_ref      = sidecar.validator_image_ref (must == configured)
validator_image_digest   = sidecar.validator_image_digest (must == configured)
inspected_image_digest   = sidecar.inspected_image_digest (must == validator_image_digest)
inspection_hash          = sidecar.inspection_hash
container_runtime        = sidecar.container_runtime
runtime_version          = sidecar.runtime_version
network_disabled         = sidecar.network_disabled (must be true)
non_root                 = sidecar.non_root (must be true)
read_only_rootfs         = sidecar.read_only_rootfs (must be true)
resource_limits          = sidecar.resource_limits
sidecar_report_hash      = sidecar.sidecar_report_hash (NEW receipt field)
command_results          = sidecar.command_results
aggregate_exit_status    = sidecar.aggregate_exit_status
validator_result_status  = sidecar.overall (must be "pass" for a pass receipt)
```

The app-side pass-capable derivation (Task 7 of Gap 1) is unchanged in shape:

```text
backend.supports_pass === true
  AND kind is structurally pass-capable (container-runtime)
  AND a live probe confirms the sidecar is reachable
  AND validator image identity is complete
  AND sidecar's reported isolation flags all match expected_executor_policy
  AND sidecar's inspected_image_digest == configured validator_image_digest
```

The last two conditions are NEW (they don't exist for the in-process
docker-executor because the app was the executor). The verifier gate
(`validateGap1ValidatorBindings`) continues to enforce checks 3f-3j on the
resulting receipt.

## Compose shape

```yaml
gitwire-executor-sidecar:
  build:
    context: .
    dockerfile: packages/executor-sidecar/Dockerfile
  restart: unless-stopped
  expose:
    - "3003"                    # private — NOT published to host
  environment:
    NODE_ENV: production
    PORT: 3003
    GITWIRE_VALIDATOR_IMAGE_REF: ${GITWIRE_VALIDATOR_IMAGE_REF}
    GITWIRE_VALIDATOR_IMAGE_DIGEST: ${GITWIRE_VALIDATOR_IMAGE_DIGEST}
    GITWIRE_EXECUTOR_SIDECAR_TOKEN: ${GITWIRE_EXECUTOR_SIDECAR_TOKEN}
    # NOTE: no host docker.sock mount — Docker-in-LXC works natively (Task 0).
    # The sidecar runs dockerd internally OR shares the CT-115 daemon via
    # the compose network. Decision deferred to Task 2 implementation.
  networks:
    - gitwire-net
  deploy:
    resources:
      limits:
        memory: 512M
        cpus: "1.0"
```

`gitwire-app` gains only:

```yaml
environment:
  GITWIRE_EXECUTOR_BACKEND: sidecar-executor
  GITWIRE_EXECUTOR_SIDECAR_URL: http://gitwire-executor-sidecar:3003
  GITWIRE_EXECUTOR_SIDECAR_TOKEN: ${GITWIRE_EXECUTOR_SIDECAR_TOKEN}
```

No `volumes:` mount of `/var/run/docker.sock` on `gitwire-app`. The app
container's Docker authority remains zero.

## v0.23.0 acceptance criteria

CT 115 `/health` moves from the v0.22.0 state to:

```text
executor.selected_kind          = container-runtime
executor.selected_pass_capable  = true
validator.configured            = true
validator.pass_capable          = true
validator.reason                = configured_and_pass_capable
```

A pass receipt produced via the sidecar contains:

```text
execution_backend_id     = sidecar-executor
executor_kind            = container-runtime
executor_pass_capable    = true
validator_image_ref      = <configured, digest-pinned>
validator_image_digest   = <configured>
inspected_image_digest   = <must equal validator_image_digest>
inspection_hash          = <present>
container_runtime        = docker (or podman)
runtime_version          = <detected>
network_disabled         = true
non_root                 = true
read_only_rootfs         = true
resource_limits          = <present>
sidecar_report_hash      = <present>
validator_result_status  = pass
```

Failure modes that MUST yield `inconclusive` or `blocked`, never `pass`:
- sidecar unreachable
- validator image identity not configured
- sidecar's inspected digest ≠ configured digest
- sidecar reports any isolation flag as false
- sidecar's `overall` is inconclusive or fail
- any command's exit_status is null (incomplete execution)

## Non-goals

This design does **not**:

- Implement a delegated-run provider (Gap 1 Phase 4 contract remains untouched).
- Bind the host Docker socket into `gitwire-app`.
- Make `local-process` pass-capable (the v0.22.0 invariant holds).
- Weaken any Gap 1 verifier check (3f-3j remain enforced).
- Change the existing `docker-executor` backend (kept for non-sidecar envs).
- Address multi-tenant execution (deferred — delegated-run is the future path).
- Perform post-apply proof (Gap 3, future).
