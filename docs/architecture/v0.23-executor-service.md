# Container-Runtime Executor Service (v0.23.0)

## Status

Design pass, revision 3. Rev 1 introduced the sidecar concept; rev 2 reframed
"sidecar" → "executor service" (contract vs deployment topology); rev 3
resolves three contract-level gaps from review:

- **Runtime endpoint** is now concrete: CT-local Docker socket mounted into
  the executor service only (not the app; no nested dockerd; no TCP daemon).
- **Backend-level reachability** is now a load-bearing requirement: pass-
  capability derivation keyed by `backend_id`, not kind (the current kind-keyed
  logic in `sandboxRunner.js` is unsafe once two backends share
  `container-runtime`).
- **`executor_report_hash` verifiability**: the app must persist raw executor
  report content (or a content-addressed ref) so the verifier can recompute
  the hash. A bare hash in the receipt is not sufficient.

Task 0 (CT 115 feasibility) is GREEN. No implementation code yet; this document
defines the contract for review before Task 2+.

Companion to: `docs/architecture/validator-execution-model.md` (Gap 1, v0.22.0).

## The actual requirement

```text
Give GitWire a pass-capable execution boundary
without giving the app container Docker authority,
and bind the resulting proof into receipts.
```

Everything else — "sidecar", "remote executor", "VM executor" — is a deployment
of that boundary, not the boundary itself. This document defines the boundary
as a contract (the **executor service**) and names the first deployment of it
(the **compose-local service** on CT 115).

## Why not name the architecture "sidecar"

A sidecar is a deployment pattern: a companion process placed next to a primary
service to provide one narrow capability. Naming the v0.23.0 architecture
"sidecar executor" would bake the topology into the product model — forcing a
rename when the same contract is deployed to a remote host, a VM, or when a
second deployment is added alongside the first.

The product abstraction is the **executor service contract**. The CT 115
deployment of it happens to be a compose sibling ("sidecar" in the loose
sense). Future deployments of the same contract include remote executor hosts,
VM/microVM executors, and (as a separate provider model) delegated-run.

## Architectural framing

```text
GitWire app:
  policy, DB, GitHub credentials, final receipt construction, verifier gate

Executor service (contract):
  isolated validator execution, runtime inspection, raw execution report

Executor driver / deployment mode (instance property of the service):
  compose-local service   # v0.23.0 target for CT 115
  remote executor host    # future
  VM / microVM            # future
  delegated-run provider  # future, SEPARATE kind (EXECUTOR_KINDS.DELEGATED_RUN)
```

The app should not care whether the executor service is a compose sibling, a
remote host, or a VM. It cares only about:

```text
backend_id
backend_kind
reachability
authentic executor report
validator image identity
runtime isolation evidence
receipt binding
```

## Context

v0.22.0 made the validator execution topology honest. On CT 115 it now reports:

```text
executor.selected_kind          = local-process
executor.selected_pass_capable  = false
validator.pass_capable          = false
validator.reason                = selected_backend_not_pass_capable
```

This is correct but not the end goal: CT 115 cannot *produce* pass-capable
validator evidence. `local-process` is structurally non-pass-capable, and the
existing `docker-executor` backend — while `supports_pass: true` — is not
safely reachable from the app container on CT 115 (the app process should not
own Docker authority; binding the host socket into the app container is
explicitly disallowed by the Gap 1 decision).

v0.23.0 introduces an **executor service** that owns container-runtime access
and runs the pinned validator image on behalf of the app. The initial
deployment is a compose-local service on CT 115; the contract is designed to
survive being moved to a remote host or VM later without renaming.

No privilege narrowing needed.

### Key finding: Docker-in-LXC already works

Docker runs *inside* CT 115 (the LXC container), using the Proxmox host's
kernel via cgroup v2 delegation. This means the executor service can reach a
real Docker daemon without the app ever touching Docker authority.

### Runtime endpoint (v0.23.0 default — resolved)

The v0.23.0 default is **concrete, not deferred**:

```text
Mount the CT-local Docker Unix socket (e.g. /var/run/docker.sock) into
gitwire-executor-service ONLY. Do NOT mount it into gitwire-app.
Do NOT run a nested dockerd as the v0.23.0 path.
Do NOT expose a TCP Docker daemon.
```

Rationale:
- The Unix-socket-into-one-container pattern preserves the core boundary: the
  app has zero Docker authority; only the executor service has runtime
  authority, and only through its narrow validation API.
- Running `dockerd` inside the executor service is a *future hardening* path.
  It likely requires additional cgroup/device/privilege analysis that does not
  belong in v0.23.0.
- A TCP Docker daemon is strictly more exposed than a mounted Unix socket and
  is rejected unless paired with a tightly scoped socket-proxy or mTLS design
  (also future).

The compose sketch below reflects this default (`volumes:` on the executor
service, not on the app).

## Options evaluated (and why executor service won)

### Option A — Mount Docker into the app
`gitwire-app → /var/run/docker.sock → validator container`.

Simplest technically, but **reject**: it collapses the separation we just
built. The app has DB credentials, GitHub credentials, and policy authority;
giving it container-runtime authority on top is exactly the consolidation the
Gap 1 decision forbids.

### Option B — Container-runtime executor service (RECOMMENDED for v0.23.0)
`gitwire-app → narrow private API → executor service → Docker/Podman → validator`.

Keeps Docker authority out of the app, reuses the existing container-runtime
isolation model, shortest path to `validator.pass_capable = true`. The
contract is topology-neutral; the first deployment is a compose-local service.

### Option C — Delegated-run provider
`gitwire-app → external execution provider → validator proof`.

Better long-term for multi-tenant scaling, but introduces provider trust,
provider receipts, network dependency, cost, and vendor choice before we have
proven the local executor-service contract. The Gap 1 Phase 4
`delegatedRunProvider.js` contract stays in place untouched for this future
path.

### Option D — GitHub Actions as validator backend
Tempting (GitHub already has runners, logs, artifacts), but awkward proof
semantics: network access, GitHub token surfaces, runner variability, and it
blends "CI" and "GitWire proof" which should remain distinct. Not ideal for
the proof boundary.

### Option E — Dedicated VM/microVM executor
Architecturally strong — even cleaner boundary than a compose sibling — but
operationally heavier and premature for v0.23.0. Excellent future hardening
path; not the first implementation.

## The executor service contract

The service exposes exactly two endpoints. It is deliberately **not** a general
Docker API — callers cannot name arbitrary images or run arbitrary commands.

### `GET /health`

```json
{
  "status": "ok",
  "executor_service_id": "executor-service",
  "executor_service_version": "1.0.0",
  "executor_service_instance_id": "<random per-instance uuid>",
  "deployment_mode": "compose-local",
  "container_runtime": "docker",
  "runtime_version": "29.5.0",
  "validator_image_ref": "<configured>",
  "validator_image_digest": "<configured>",
  "ready": true
}
```

`ready` is true only when: runtime reachable, validator image identity
configured, and image inspection has succeeded at least once. `deployment_mode`
is reported by the instance (`compose-local`, `remote`, `vm`) — informational;
the app does not branch on it.

### `POST /v1/validate`

**Authenticates** via a shared bearer token (`GITWIRE_EXECUTOR_SERVICE_TOKEN`)
set identically in both the app and executor service environments. Not a
substitute for the network boundary — the service must only be reachable on
the private network, never exposed externally.

**Request:**

```json
{
  "request_id": "<uuid, for tracing>",
  "files": [
    { "path": "src/foo.js", "content": "..." }
  ],
  "commands": ["lint", "test", "build", "typecheck"],
  "limits": {
    "wall_clock_ms": 30000,
    "memory_mb": 512,
    "pids_limit": 64,
    "output_bytes": 1048576
  },
  "validator_image_ref": "<registry/repo/image@sha256:...>",
  "validator_image_digest": "sha256:...",
  "expected_executor_policy": {
    "network_disabled": true,
    "non_root": true,
    "read_only_rootfs": true,
    "resource_limits": true
  }
}
```

**Response (success path):**

```json
{
  "report_schema_version": 1,
  "executor_service_id": "executor-service",
  "executor_service_version": "1.0.0",
  "executor_service_instance_id": "<uuid>",
  "deployment_mode": "compose-local",
  "container_runtime": "docker",
  "runtime_version": "29.5.0",
  "validator_image_ref": "<full ref>",
  "validator_image_digest": "sha256:...",
  "inspected_image_digest": "sha256:...",
  "inspection_hash": "sha256:...",
  "network_disabled": true,
  "non_root": true,
  "read_only_rootfs": true,
  "resource_limits": { ... },
  "command_results": [
    {
      "command": "lint",
      "exit_status": 0,
      "output_ref": "output:sha256:...",
      "output_hash": "sha256:...",
      "duration_ms": 1234
    }
  ],
  "aggregate_exit_status": 0,
  "overall": "pass",
  "executor_report_hash": "sha256:<over the full response object minus this field>",
  "executor_report_ref": "executor-report:sha256:<same hash>"
}
```

`report_schema_version` lets the verifier and the persisted-store logic evolve
the report shape without ambiguity. `executor_report_ref` is the
content-addressed reference the app uses when persisting the raw report (see
"Verifier requirements" below).

**Response (failure / inconclusive paths):**

```json
{
  "overall": "inconclusive",
  "inconclusive_reason": "validator_image_ref_mismatch | image_inspection_failed | runtime_unreachable | execution_incomplete | executor_error",
  "inconclusive_detail": "...",
  "executor_service_id": "executor-service",
  "executor_service_version": "1.0.0"
}
```

The service never returns `overall: "pass"` unless it has actually run all
commands against the pinned image AND every isolation flag was enforced AND
the inspected image digest matches the configured digest. The app re-verifies
this (defense in depth — see Receipt Binding below).

### Repo-aware command descriptors (Task 8D)

The fixed command IDs (`lint → npm run lint --`) validate the **GitWire
baseline's** `packages/*/src/`, not the healed repo's files. Task 8D adds
repo-aware command descriptors derived from CI evidence so the verifier runs
the target repo's actual command (e.g. `npx --no-install eslint app.js`).

**Derivation.** At CI-evidence collection time, the workflow YAML is fetched
**pinned to `head_sha`** (never the default branch) and parsed. Safe
`run:` invocations are compiled into descriptors and frozen into a
`ci_workflow_command` evidence ref:

```json
{
  "type": "ci_workflow_command",
  "source": ".github/workflows/ci.yml@<head_sha>",
  "workflow_path": ".github/workflows/ci.yml",
  "workflow_ref": "<head_sha>",
  "workflow_blob_sha": "<contents-api sha>",
  "descriptor_hash": "<sha256 over the descriptor>",
  "descriptor": {
    "semantic_id": "lint_result",
    "command_id": "repo_lint",
    "source": "ci_workflow",
    "argv": ["npx", "--no-install", "eslint", "app.js"],
    "target_paths": ["app.js"],
    "network": "disabled",
    "requires_shell": false,
    "policy_status": "pending_executor_validation"
  }
}
```

**Plan compilation.** When a descriptor is present, the validation-plan
adapter uses it **in place of** the legacy `lint`/`test`/`build` template.
The plan carries a keyed `command_descriptors` object alongside `commands[]`:

```json
{
  "commands": ["repo_lint"],
  "command_descriptors": {
    "repo_lint": { "argv": ["npx", "--no-install", "eslint", "app.js"], "..." : "" }
  }
}
```

Descriptors are canonicalized via the **shared** `@gitwire/core` helper so
`sandboxRunner.buildValidationPlan()` and
`repairProposalService.buildValidationPlanForRecorder()` produce byte-identical
`validation_plan_hash` values (the P0 equivalence invariant).

**Policy enforcement (fail-closed).** The web side validates **shape only**
via `@gitwire/core`. The executor-service is the **authoritative** security
boundary — `enforceDescriptorPolicy` rejects, fail-closed, any descriptor that:

```
has requires_shell !== false
has network !== "disabled"
uses a binary outside {npx, node, tsc}       (npm run is NEVER a descriptor)
runs npx without --no-install immediately after
contains shell metacharacters in any argv element
has target_paths that are empty, absolute, contain .. , or contain globs
```

A present-but-invalid descriptor becomes an **explicit rejected artifact** —
it is never silently dropped and never falls back to the legacy template:

```json
{
  "command": "repo_lint",
  "status": "rejected",
  "command_source": "ci_workflow",
  "policy_reasons": ["npx requires --no-install"],
  "exit_status": null
}
```

A rejected `command_result` forces `overall != "pass"`.

**Receipt fields.** Each `command_result` records what executed:

```json
{
  "command": "repo_lint",
  "semantic_id": "lint_result",
  "command_source": "ci_workflow | fallback_template",
  "executed_argv": ["npx", "--no-install", "eslint", "app.js"],
  "target_paths": ["app.js"],
  "exit_status": 0
}
```

Legacy fixed command IDs remain as fallback **only when no descriptor is
present**. This is what unblocks Task 8 Step 5B: the MyShell proof can now
execute `npx --no-install eslint app.js` against the patched `app.js` instead
of `npm run lint` against the GitWire baseline.

### `executor_report_hash` semantics + verifiability

The service computes a content-addressed hash over its full response object
(minus the hash field itself), mirroring GitWire's execution-receipt pattern.
The app binds this hash into the GitWire execution receipt so the verifier
gate can confirm the receipt was built from a specific, unmodified executor
report — not an arbitrary response the app constructed on its own.

**A bare hash in the receipt is not sufficient.** For the verifier to confirm
the receipt was actually built from a real executor report — and not from an
arbitrary hash value the app (or an attacker who compromised the app) wrote —
the verifier must be able to **recompute** the hash from durable raw report
material. This is the same write-once content-addressing property GitWire's
own execution receipts rely on (in `executionReceiptStore.js`).

#### Required persistence (v0.23.0)

The app MUST persist the executor report durably before constructing the
GitWire execution receipt. Two acceptable shapes:

```text
Shape A (raw content):
  executor_report_content  = the canonical JSON the service returned
  executor_report_hash     = sha256 over (content minus the hash field)
  verifier recompute       = parse content, drop hash field, recompute, compare

Shape B (content-addressed ref):
  executor_report_ref      = "executor-report:sha256:<hash>"
  stored via executionReceiptStore (write-once, content-addressed)
  verifier resolve         = resolveReceipt(executor_report_ref), recompute hash, compare
```

Either shape works. Shape B reuses the existing `executionReceiptStore`
machinery (write-once, `ON CONFLICT DO NOTHING`, content-addressed); Shape A
is simpler but needs its own table/column. **Recommended for v0.23.0: Shape B**
— it does not require a new migration (executor reports ride inside the
existing opaque-JSON `execution_receipts` table, typed by ref prefix).

#### Verifier requirements

Before accepting a pass receipt built from an executor-service report, the
verifier MUST:

```text
1. Resolve executor_report_ref → raw report content (or read executor_report_content)
2. Recompute executor_report_hash from the raw report (minus the hash field)
3. Confirm recomputed hash == receipt's executor_report_hash
4. Confirm the receipt's execution_backend_id == report's executor_service_id
5. Confirm report.overall == "pass"
6. Confirm report's isolation flags all match expected_executor_policy
7. Confirm report.inspected_image_digest == configured validator_image_digest
```

If any of these fail, the receipt is rejected — never `pass`. This closes the
"app could write an arbitrary hash" hole: a receipt with no durable raw
report, or whose raw report's recomputed hash differs, is unverifiable and
therefore not pass-capable.

Asymmetric signing (HMAC or signature over the report) is **explicitly
deferred** — persisted raw report + recomputed hash is sufficient for v0.23.0.
Signing becomes interesting when the app and the executor service are
operated by different trust principals; that's a future-hardening item, not
v0.23.0.

**Naming note:** the field is `executor_report_hash`, not `sidecar_report_hash`.
The hash is over an executor-service report; the deployment mode does not
rename it. A future remote-host or VM deployment of the same contract uses the
same field name.

## Security boundary

The executor service is privileged *relative to the app* (it can create
containers), so its API must be narrower than Docker. Enforced invariants:

```text
Allowed:
  - run ONLY the configured validator image (request ref must match config)
  - mount ONLY the disposable workspace provided in the request
  - execute ONLY allowlisted command IDs (resolved via argv templates)
    OR repo-aware descriptors that pass enforceDescriptorPolicy (Task 8D)
  - enforce network=none, read-only rootfs, non-root, resource limits
  - return evidence

Forbidden (by design, enforced in service code):
  - arbitrary image names (validator_image_ref must equal configured)
  - arbitrary shell commands (commands[] must be in COMMAND_TEMPLATES,
    OR a descriptor argv that fails the policy gate)
  - npm run as a descriptor (it executes shell strings from package.json)
  - DB access (service has no DATABASE_URL)
  - GitHub credentials (service has no GITHUB_* env)
  - repo write access (service has no repo mounts)
  - outbound network inside the validator container (--network=none)
```

This mirrors the existing `dockerExecutorBackend` isolation contract; the
executor service is essentially `dockerExecutorBackend` re-homed behind a
private API and given its own identity.

### Network exposure

The executor service MUST NOT be exposed outside its deployment network. For
the v0.23.0 compose-local deployment:
- No port published to the host (no `ports:` mapping, only `expose:`).
- The Cloudflare tunnel (`gitwire-tunnel-1`) routes only to `gitwire-app`,
  never to the executor service.
- The shared bearer token is a second layer of defense, not the primary one.

For future remote-host deployments, the network-exposure question is different
(TLS, mTLS, network ACLs) but the principle is the same: the executor service
is not a public API.

### Receipt binding (the load-bearing invariant)

The app constructs the final GitWire execution receipt from the executor
service's evidence — the service does not write receipts. The receipt binds:

```text
execution_backend_id     = "executor-service"
executor_kind            = "container-runtime"
executor_pass_capable    = true   (only when all conditions hold)
validator_image_ref      = service.validator_image_ref (must == configured)
validator_image_digest   = service.validator_image_digest (must == configured)
inspected_image_digest   = service.inspected_image_digest (must == validator_image_digest)
inspection_hash          = service.inspection_hash
container_runtime        = service.container_runtime
runtime_version          = service.runtime_version
network_disabled         = service.network_disabled (must be true)
non_root                 = service.non_root (must be true)
read_only_rootfs         = service.read_only_rootfs (must be true)
resource_limits          = service.resource_limits
executor_report_hash     = service.executor_report_hash   # NEW receipt field
command_results          = service.command_results
aggregate_exit_status    = service.aggregate_exit_status
validator_result_status  = service.overall (must be "pass" for a pass receipt)
```

The app-side pass-capable derivation (Task 7 of Gap 1) is **augmented** for
the executor-service backend (kind-level derivation is unsafe once two
backends share `container-runtime` — see "Backend-level reachability" below):

```text
backend.supports_pass === true
  AND kind is structurally pass-capable (container-runtime)
  AND a live probe confirms the SELECTED BACKEND (by backend_id) is reachable
  AND validator image identity is complete
  AND service's reported isolation flags all match expected_executor_policy
  AND service's inspected_image_digest == configured validator_image_digest
  AND service's selected_backend_id == app's execution_backend_id (consistency)
```

The last three conditions are NEW (they don't exist for the in-process
docker-executor because the app was the executor and there was only one
backend per kind). The verifier gate (`validateGap1ValidatorBindings`)
continues to enforce checks 3f-3j on the resulting receipt; the additional
backend_id + executor_report_hash checks live at the verifier layer too
(see "Verifier requirements" below).

## Backend-level reachability (load-bearing)

**This is a required code amendment in Task 3/4 — not optional.** The current
receipt-derivation logic in `sandboxRunner.js` (lines ~297-302) computes
`reachableKinds` from probes and then checks `reachableKinds.has(executorKind)`:

```js
const probe = probeAllBackends();
const reachableKinds = new Set(
  probe.backends.filter((b) => b.reachable).map((b) => b.kind)
);
const kindActuallyReachable = reachableKinds.has(executorKind);
```

This is **only safe today because exactly one backend (`docker-executor`) maps
to `container-runtime`.** The moment `executor-service` is registered alongside
it, this check would pass whenever *either* backend was reachable — even if
the *selected* one was down. A pass-capable receipt could be built against an
unreachable selected backend.

Required amendment (Task 3/4):

```text
Reachability for receipt derivation MUST be keyed by backend_id, not kind:
  selected_backend_id = backend.id                          (the chosen backend)
  reachable_backends   = Set of backend_ids that probed reachable
  backend_reachable    = reachable_backends.has(selected_backend_id)
  executor_pass_capable = ... AND backend_reachable AND ...
```

Public `/health.executor.summary` may remain kind-keyed for operator
readability, but MUST additionally expose:

```text
executor.selected_backend_id       # which backend was selected
executor.selected_backend_reachable # whether THAT backend (not its kind) is reachable
```

This keeps the externally-observable kind summary stable (so dashboards don't
break) while making the proof-granularity signal explicit.

### Identifier consistency rule

A pass receipt must additionally enforce:

```text
execution_backend_id (in receipt) == selected_backend_id (from /health probe)
                                     == executor_service_id (in service response)
```

If any of these disagree, the receipt must be rejected — the app cannot
construct pass-capable evidence against a service response that did not come
from the backend the app selected and probed.

## Deployment modes

The contract is the same across deployment modes. The instance reports its
mode via `deployment_mode` in `/health` and `/v1/validate` responses for
observability. **`deployment_mode` is observability-only — it MUST NOT
participate in pass-capability derivation, verifier branching, or any policy
decision.**

```text
Allowed uses of deployment_mode:
  - show in /health response
  - include in executor report
  - surface in operator dashboards / logs for debugging

Forbidden uses of deployment_mode:
  - pass predicate (a remote deployment is not inherently more or less
    pass-capable than a compose-local one — pass-capability comes from the
    four+ conditions in Receipt Binding, not from topology)
  - branching verifier behavior
  - making policy decisions
```

The app must not branch on `deployment_mode`. Pass-capability follows from
backend reachability, validator identity, isolation evidence, and report
verifiability — all of which are deployment-mode-agnostic by design.

### Task 8 Step 5B — Automated pass receipt: BLOCKED (architectural)

**Blocker:** The validator image's command templates are hardcoded to
GitWire's own project structure. The `lint` command (`npm run lint --`)
resolves to `eslint packages/*/src/` from the GitWire baseline, NOT the
healed repo's files. The patched `app.js` is overlaid onto the workspace,
but the lint command ignores it entirely.

This means no patch can produce a lint pass receipt through the current
executor-service, because the lint command validates GitWire's source
(which has lint errors), not the target repo's patched files.

**Root cause:** The command template system
(`validatorRunner.js:COMMAND_TEMPLATES`) uses fixed argv templates:

```js
const COMMAND_TEMPLATES = Object.freeze({
  lint:      ["npm", "run", "lint", "--"],
  test:      ["npm", "test", "--"],
  build:     ["npm", "run", "build", "--"],
  typecheck: ["npm", "run", "typecheck", "--"],
});
```

These resolve `npm run lint` against the validator image's `package.json`,
which is GitWire's baseline. The actual CI failure was `npx eslint app.js`
(MyShell's workflow), but the validator runs GitWire's lint instead.

**This is NOT a patch-correctness bug.** The patch generator, artifact
applier, receipt binding, hash recompute, digest gate, and fingerprint gate
all work correctly. The blocker is in the command template layer.

## Normative Design Amendment — Plan-Execution Conformance Model

This section documents the new rules introduced by the plan-execution
conformance PR. It is not the full rev 4 reconciliation (governance ledger,
chronology, supersession matrix) — that is a separate task. This amendment
covers only the rules the enforcement PR implements.

### Plan schema v2

The validation plan now carries an explicit normative-step list:

```json
{
  "plan_schema_version": 2,
  "descriptor_policy": { "activation": "observed" },
  "required_execution_features": ["normative-step-reporting-v1"],
  "normative_steps": [
    {
      "step_id": "lint_result:0",
      "sequence": 0,
      "semantic": "lint_result",
      "command_source": "legacy_template",
      "command_id": "lint",
      "argv": null,
      "target_paths": null
    }
  ],
  "executable_commands": ["lint"],
  "command_descriptors": {}
}
```

`normative_steps` is the source of truth for what the plan requires.
`executable_commands` is a derived backward-compat projection.

### Observed vs selected semantics

`GITWIRE_DESCRIPTOR_ACTIVATION=observed|selected` (default `observed`):

- **`observed`**: descriptors are derived and frozen as evidence candidates.
  Legacy commands are explicitly selected as normative. The plan hash binds
  the legacy selection. Descriptors remain in `command_descriptors` for
  auditability but are not executed.
- **`selected`**: valid descriptors become normative. `command-descriptor-v1`
  is added to `required_execution_features`. Invalid descriptors are
  fail-closed (no legacy fallback). An applicable-but-invalid descriptor
  blocks descriptor execution; legacy is not silently selected.

Invalid configuration value → startup failure (no silent coercion).

### Exact conformance requirement

A receipt can authorize `pass` or `fail` only when `plan_execution_relation`
is `exact` — every normative step has exactly one matching executed step
with identical argv (element-by-element), matching target_paths (normalized),
matching command_source, and no extra or omitted steps.

The app derives this independently — it does NOT trust a relation string
from the executor. The verifier recomputes from the canonical plan + the
raw executor report (or receipt `executed_steps` for non-executor-service
backends) and enforces `recomputed == exact` AND
`receipt.plan_execution_relation == recomputed`.

### Execution relation values

- **`exact`**: every normative step matched exactly.
- **`none`**: trustworthy evidence establishes no normative step ran.
- **`divergent`**: at least one step was attempted/omitted/substituted/
  duplicated, or executed with different argv or targets.
- **`unverifiable`**: available evidence is insufficient to determine
  exact, none, or divergent.

Both `none` and `unverifiable` yield `inconclusive`. `divergent` also
yields `inconclusive` (not `fail` — an unplanned command must not reject
a repair).

### Backend feature contract

Backends declare `execution_features`:
- `normative-step-reporting-v1`: backend produces structured `executed_steps`.
- `command-descriptor-v1`: backend can execute descriptor argv after policy
  validation.

Descriptor support is NOT inferred from backend ID, kind, reachability, or
`supports_pass`. In `selected` mode, missing `command-descriptor-v1` makes
the result ineligible (not `divergent` — capability is separate from factual
relation).

Backend features are frozen into the receipt. Historical verification uses
the receipt-bound snapshot, not the current backend registry.

### Result eligibility rule

```
authoritative_result_eligible =
    plan_execution_relation == exact
    AND required_features ⊆ backend_execution_features
    AND backend_pass_capable
    AND selected_backend_reachable
    AND validator_image_identity_valid
    AND report_integrity_valid
    AND execution_evidence_complete
    AND plan_schema_supported
```

Result mapping:
- `eligible + success → pass`
- `eligible + failure → fail`
- `ineligible → inconclusive`
- `timeout/spawn failure → inconclusive`

### Historical schema interpretation

Plan schema v1 receipts (before `plan_schema_version: 2`) lack
`normative_steps` and structured `executed_steps`. They cannot establish
`plan_execution_relation = "exact"` and are classified `unverifiable` under
the conformance model. Their `validation_plan_hash` remains valid as a
historical artifact (the schema-v1 hash algorithm is preserved separately).

### Invariants

1. Empty plans cannot pass vacuously (zero steps + zero executed → `none`).
2. Step IDs are validated end-to-end (duplicate/unknown/missing → `divergent`).
3. Backends must capture the actual argv passed to the process API — not
   reconstruct by splitting command strings. Shell execution is represented
   explicitly or classified `unverifiable`.
4. Relation is separate from outcome (`exact` + failure → `fail`;
   `exact` + timeout → `inconclusive`).
